Build a rolling 13-month cash flow forecasting web app for a business that already reconciles to actual bank cash position.

The model must use actual cash as the anchor, not just accounting accruals.

Core objective

Create a system that:

imports actual cash movements from Xero as the primary source of truth

reconciles to two bank accounts

ensures all historic and current-day transactions are captured so the model is 100% reconciled to actual cash

shows:

actual current month

12 future months forecast

automatically updates future forecast months when actual recurring costs change, such as supplier price increases or contractual uplifts

calculates total free cash flow generated by the business

The current spreadsheet logic is mostly line by line / row by row, with months in columns, but the app should not use spreadsheet columns as the core data structure. Instead, store data in a proper database and render the output in a spreadsheet-like monthly view.

Functional requirements
1. Data source and actuals

Use Xero as the principal source of data.

Import and sync:

bank transactions

spend money / receive money transactions

invoices and bills where useful

payments

bank transfers

account balances if available

transaction IDs and references for audit trail

The system must:

map all actual historical and current transactions into cash flow lines

assign each transaction to one of the two bank accounts

reconcile the total cash position to actual bank cash

prevent forecast values from remaining in closed or actualised periods once actual data is known

Historic periods should be locked as actuals once reconciled.

2. Forecast structure

The model should show:

current actual month

next 12 months forecast

Each cash flow line should support one of these behaviours:

predictable recurring fixed

predictable recurring with annual uplift

recurring but timing-variable

semi-variable recurring

one-off

Examples:

payroll

rent

insurance

software subscriptions

compliance costs

annual renewals

irregular but recurring maintenance

one-off exceptional costs

Some lines are fully detailed. Some lines are rolled up. Both detailed and rolled-up lines must still form part of the full reconciled cash model.

3. Forecast rebasing logic

This is critical.

When actuals arrive, the system must compare actual cash movement against the original forecast.

For each variance, the system must classify it as one of:

timing difference

permanent run-rate change

one-off variance

Behaviour:

timing difference: move between months only, do not change future baseline

permanent run-rate change: update the baseline and flow the revised amount into all relevant future forecast months

one-off variance: affect current period only, do not flow into future months

Example:

forecast software cost = 1,000 per month

actual supplier invoice arrives at 1,150

if marked as permanent price increase, all future months update to 1,150

if marked as timing difference, only the affected month changes

if marked as one-off setup fee, only that month changes

The app must allow a user to approve, override, or correct the system’s treatment.

4. User workflow

The workflow should be:

Sync actuals from Xero

Match/map transactions to cash flow lines

Update current month actuals

Show variances vs prior forecast

Suggest variance treatment

Allow user to approve or override

Flow approved changes through future months

Recalculate closing cash and free cash flow

5. Key outputs

The main output screen should be a spreadsheet-style monthly cash flow view with:

rows = cash flow lines

columns = actual current month + next 12 months

subtotals for major categories

totals for cash inflows

totals for cash outflows

net monthly cash movement

opening cash

closing cash

free cash flow

Also include:

actual vs forecast variance by line

variance classification

audit history of changes

confidence flag on forecast lines

indication of which months are actual and which are forecast

Free cash flow

The app should calculate free cash flow generated by the business.

At minimum, support:

operating cash inflows

operating cash outflows

recurring compliance and annual costs

optional separation of:

capex

financing

tax

one-offs

Show at least:

net cash movement

closing cash

free cash flow

free cash flow excluding one-offs

Forecast engine rules

Build a forecast engine that:

generates monthly forecast values for the next 12 months

applies recurrence rules

applies annual uplifts

supports monthly, quarterly, annual, and custom cycles

supports payment timing offsets

supports manual overrides

supports rebasing from actuals

Priority order for monthly forecast calculation:

manual override

approved rebase from actual variance

standard recurrence rule

default zero

Mapping logic

Implement transaction mapping logic using:

supplier name matching

keywords in description/reference

account / tracking category if available from Xero

historical mapping patterns

manual correction by user

The system may use AI assistance to suggest mappings, but user corrections must be stored and used as future rules.

Do not rely solely on AI. This is a finance control tool, so deterministic rules and auditability are required.

AI-assisted features

Use AI only as a support layer, not as the core logic.

Useful AI features:

suggest transaction-to-line mapping

suggest whether a variance is:

timing

permanent run-rate change

one-off

detect likely supplier price increase patterns

flag anomalies or unusual spikes

explain why a forecast changed

All AI suggestions must be reviewable and overridable by the user.

Controls and governance

This app must behave like a finance-controlled tool.

Requirements:

every change must be auditable

actuals must not be overwritten by forecast logic

users must be able to lock historic periods

future changes must record the reason and source

no silent changes to future months

all rebasing actions must be traceable

Business rules

Apply these business assumptions:

The business reconciles to actual cash position, not just nominal ledger balances

All costs are already captured because actual cash is reconciled

The model is a rolling 13-month cash view

The current month is actualising continuously

The next 12 months are forecast

Forecast should update when:

actual recurring price changes occur

known annual uplifts occur

known contract renewals occur

timing shifts occur

manual overrides are entered

Historic actual months are never altered by forecast rules

The system must support both detailed lines and rolled-up lines

The model must remain fully reconcilable to two bank accounts

Important design instruction

Do not build this as a spreadsheet clone behind the scenes.

Use a proper database-driven model with event/rule logic, then render the output in a familiar spreadsheet-style grid for users.

Accuracy, auditability, and controlled forecast rebasing are more important than appearance.
